# OrionTV 播放源 JSON 档案化设计

## 背景

当前 OrionTV 的默认播放源直接写死在 `services/luna/config.ts` 中，应用运行时通过 `lunaConfig.getApiSites()` 读取当前源列表。现状存在几个问题：

- 默认源配置耦合在代码里，新增或替换一套源需要改代码并重新发版。
- 只能使用单一默认源集合，无法在多套配置之间切换。
- 用户无法从本地导入自己的 `LunaTV-config.json` 并长期保存为可切换档案。

本次设计的目标是把“播放源配置”从单一静态默认值升级为“可切换的 JSON 档案”，同时兼容：

- 内置多个 JSON 预设
- 从本地导入 `LunaTV-config.json`
- 将导入结果保存为用户档案并随时切换

## 目标

- 将播放源配置建模为 `source profile`，而不是单个 `apiSites`
- 支持内置预设档案与外部导入档案统一管理
- 支持在设置页切换当前档案
- 支持从本地文件导入 `LunaTV-config.json`
- 支持删除已导入档案
- 保持搜索、详情、播放主链路尽量不改动，只改配置读取入口
- 对旧版本仅保存 `apiSites` 的用户数据提供自动迁移

## 非目标

第一版不包含以下能力：

- 在线 URL 导入 JSON
- 手工编辑 JSON 文本
- 导出档案
- 档案重命名
- 同时兼容多种外部 JSON 格式

## 用户场景

### 场景 1：切换内置档案

用户在设置页中看到当前正在使用的播放源档案，例如“LunaTV 默认源”。进入档案列表后，可以切换到另一个内置预设。切换完成后，播放源列表立即刷新。

### 场景 2：导入本地 JSON

用户从设备本地选择一个 `LunaTV-config.json`。应用解析并校验后，创建一个新的用户档案，默认使用文件名作为档案名，并自动切换到该档案。

### 场景 3：删除导入档案

用户删除某个已导入档案。如果该档案不是当前档案，直接删除；如果它正处于激活状态，则删除后自动回退到默认内置档案。

## 数据模型

### 现状

当前配置模型核心是：

- `apiSites: ApiSite[]`
- 其他 Luna 配置项，例如 `searchMaxPage`

### 目标模型

在配置层引入新的档案结构：

```ts
interface SourceProfile {
  id: string;
  name: string;
  type: "builtin" | "imported";
  sites: ApiSite[];
  importedAt?: number;
}

interface AppConfig {
  sourceProfiles: SourceProfile[];
  activeSourceProfileId: string;
  customCategories: CustomCategory[];
  siteName: string;
  searchMaxPage: number;
  doubanProxyType: string;
  disableYellowFilter: boolean;
}
```

同时保留迁移兼容逻辑：

- 如果旧存储里只有 `apiSites`
- 首次读取时自动转换成一个默认内置档案
- 并写回新结构，避免每次启动重复迁移

### 建模说明

- `builtin` 用于标识应用内置 JSON 档案
- `imported` 用于标识用户从本地文件导入的档案
- `activeSourceProfileId` 表示当前实际生效的播放源档案
- `lunaConfig.getApiSites()` 只返回当前激活档案中的 `sites`

这样搜索、详情和播放流程仍然只关心 `ApiSite[]`，无需感知档案管理的存在。

## JSON 预设设计

### 内置档案来源

内置播放源从代码常量中抽离，改为独立 JSON 预设文件或映射模块。第一版至少包含一个默认档案：

- `LunaTV 默认源`

结构上预留多个内置预设入口，便于后续继续增加：

- `精简版`
- `家庭版`
- `测试版`

### 预设加载原则

- 应用启动时始终能构造出完整的内置档案列表
- 内置档案不可删除
- 内置档案名称固定，由应用侧控制

## 外部导入设计

### 仅支持的外部格式

第一版只支持 `LunaTV-config.json` 结构：

```json
{
  "cache_time": 7200,
  "api_site": {
    "example.com": {
      "name": "示例资源",
      "api": "https://example.com/api.php/provide/vod",
      "detail": "https://example.com"
    }
  }
}
```

### 导入转换规则

导入时将 `api_site` 下的每个条目转换为项目内部的 `ApiSite`：

- `key`：根据域名或原始键名标准化生成
- `name`：直接取配置中的 `name`
- `api`：直接取配置中的 `api`
- `detail`：直接取配置中的 `detail`

### 校验规则

- 文件不是合法 JSON：导入失败，提示“JSON 解析失败”
- 顶层不存在 `api_site` 或 `api_site` 不是对象：导入失败，提示“不是支持的 LunaTV 配置格式”
- 单个站点缺少 `name` 或 `api`：跳过该站点并统计数量
- 全部站点无效：整次导入失败，不创建档案
- 同一文件内站点 key 冲突：后出现的覆盖前一个
- 档案名冲突：自动追加序号，例如 `我的源 (2)`

### 导入后行为

- 导入成功后创建一个新的 `imported` 档案
- 档案名默认取文件名（去掉扩展名）
- 导入成功后自动切换到新档案
- 同时触发资源刷新，使设置页中的源列表立即变化

## 设置页交互设计

第一版在设置页引入一个轻量档案管理区，保持 TV 遥控器交互尽量简单。

### 展示内容

- 当前档案名
- 档案切换入口
- 导入 JSON 入口
- 删除导入档案入口

### 交互能力

#### 1. 当前档案显示

展示当前正在使用的档案名，例如：

- `LunaTV 默认源`
- `我的备份源`

#### 2. 切换档案

进入列表后展示：

- 所有内置档案
- 所有已导入档案

用户选择后：

- 更新 `activeSourceProfileId`
- 重新加载资源列表
- 保持源开关区可立即感知变化

#### 3. 导入 JSON

用户从本地选择文件后：

- 读取文件内容
- 解析与校验
- 生成导入档案
- 自动切换到该档案

#### 4. 删除导入档案

约束如下：

- 只允许删除 `imported` 档案
- `builtin` 档案不显示删除能力
- 删除当前激活档案时自动切回默认内置档案

## 配置读取与刷新策略

配置层需要调整为以当前档案为准：

- `getApiSites()` 读取当前档案的 `sites`
- `addApiSite()`、`removeApiSite()`、`updateApiSite()` 如果保留，需要明确只作用于当前档案

为了避免在第一版引入行为歧义，建议：

- 现有运行期播放源开关仍然保持在 `settingsStore.videoSource`
- 它只负责“当前档案中的源是否启用”
- 切换档案后执行 `loadResources()` 并重新渲染播放源列表

如果未来需要让“编辑源”与“档案源”联动，再单独扩展。

## 迁移策略

### 旧数据识别

如果存储里读取到的是旧结构：

- 有 `apiSites`
- 没有 `sourceProfiles` 或 `activeSourceProfileId`

则判定为旧版本配置。

### 迁移过程

首次读取时：

1. 构造内置默认档案
2. 将旧的 `apiSites` 生成一个等价档案，优先作为当前档案
3. 写回新结构

### 迁移目标

- 不让旧用户因为结构升级而丢失已经保存的播放源配置
- 新老结构过渡完成后，后续逻辑统一按新结构运行

## 实现范围

第一版落地范围限定为以下模块：

### 配置层

- 扩展 `AppConfig`
- 在 `lunaConfig` 中实现档案化存储和迁移

### JSON 预设层

- 将内置源抽到独立 JSON/映射文件
- 生成内置档案列表

### 导入层

- 本地文件选择
- 文件读取
- JSON 解析
- 结构校验
- 档案生成与持久化

### 设置页

- 当前档案显示
- 切换档案
- 导入档案
- 删除导入档案

### 刷新链路

- 档案切换后刷新 `sourceStore`

## 风险与注意事项

### 1. TV 端文件选择体验

TV 端文件选择器体验通常弱于手机端，需要优先选择 Expo 或现有依赖中对 Android/TV 兼容性较好的方案。如果文件选择器在 TV 上能力不足，需要至少保证手机/平板端可用，并为 TV 给出合理兜底提示。

### 2. 旧逻辑兼容

当前搜索、详情、播放流程只认 `ApiSite[]`。必须保证 `getApiSites()` 的返回类型和行为不变，否则会引发主链路回归。

### 3. 源开关状态

当前源开关按 `resourceKey` 存储。切换档案后，旧档案里的开关状态可能不完全适用于新档案。第一版可以接受这种按 key 复用的行为，但实现时要避免因为缺失 key 导致渲染异常。

## 测试策略

### 单元测试

覆盖以下行为：

- `LunaTV-config.json -> ApiSite[]` 转换
- 无效项跳过
- 同名档案自动补序号
- 删除当前导入档案时回退默认档案

### 配置层测试

覆盖以下行为：

- 旧结构 `apiSites` 自动迁移为新档案结构
- `getApiSites()` 返回当前激活档案的源列表
- 切换档案后返回值变化正确

### adb 手测

验证以下行为：

- 清缓存后首次启动不崩溃
- 默认内置档案能正常显示
- 导入 JSON 成功后可切换为当前档案
- 切换档案后源列表发生变化

## 推荐实施顺序

1. 先实现配置层与迁移
2. 再抽离内置 JSON 预设
3. 补导入解析与档案持久化
4. 最后接设置页 UI 和交互
5. 完成后补单测与 adb 验证

## 结论

本设计采用“播放源档案化”作为统一模型，把内置预设和外部导入都收敛到同一个 `source profile` 结构中。这样既满足当前“选择使用哪个 JSON 文件”的需求，也为后续的档案管理能力留下清晰扩展点，同时尽量不扰动搜索、详情、播放主链路。
